# 针对技术规划和skills文件夹的解释

注意，请将skills和技术规划文件夹放置在项目文件夹外。

## 技术规划文件夹内容解释

这个是改完之后的技术规划档案，在总目录下有一个backend_PLAN.md，是在假设虚拟前端存在的情况下，搭建后端和其他业务模块的具体方案；PLAN.md是在不假设虚拟前端存在的情况下，搭建整个项目的具体方案；

在“底层框架+数据库、复杂独立模块专项文档、后端、前端、通用规范、业务流程”这几个文件夹下都有各个模块的PLAN.md，是各个模块搭建的具体方案，内部用引用的形式，提及了该文件夹下具体细节的文档，以便AI查阅特定文档解决问题，总的两个PLAN.md也用了这种引用的方式引用了其他模块的PLAN.md文件；

在构建每个PLAN.md的时候，特别根据“问题”文件夹下提及的问题，把解决问题的方案标注出来了，而且针对每个模块生成了需要使用的测试方法和测试代码应该放的位置

在“底层框架+数据库”里有一个”Docker接入场景下的数据库字段增补建议.md“，目前的数据库字段设计针对<接入>这一模块是够用的，但是针对docker容器的生产环境鉴别是不够的，这个是一个补齐数据库的建议；在“复杂独立模块专项文档”文件夹下，有一个“Swarmpit 接入与运维控制台专项方案.md”，这是我把一个开源项目swarmpit接入我们的项目，来监控Swarm cluster、node、service、task 的真实运行状态并进行重启、故障排查的具体实行方案，原项目开源地址为https://github.com/swarmpit/swarmpit，已将该方案并入该文件夹下的PLAN.md中

在“前端”文件夹下有一个REPORT.md，该文档介绍了当前虚拟前端暴露出的所有接口、依赖、跳转关系以及相应后端需要承担的功能，可以理解为这个虚拟前端的说明书，该文档已被并入backend_PLAN.md

## 检查点设置

在backend_PLAN.md和PLAN.md中，我设置了检查点的机制，每次完成一个阶段的任务或实现了里程碑式的验收成果，会自动建立一个checkpoint文件夹，内部会存储一个CHANGE.md 文档、一个测试日志文件夹以及这一轮改变了的代码，便于代码回滚

## skills文件夹解释

压缩包里还有一个skills文件夹，该文件夹是一个使用playwright进行联调测试的SKILL.md，参考“05-测试为什么重要，以及如何用 Playwright 联调.md”文档进行设计
